NXS vs JSON vs CSV — Browser Benchmark

The same fixtures used in the Node and Go benches, run in your browser over fetch().

What this page tests Five bar-chart workloads on the same record count, comparing JSON, CSV, and NXS side by side. (1) Open — JSON.parse / full CSV parse vs new NxsReader (header + tail-index only). (2) Warm random access — read username for a random record index. (3) Multi-field access — four fields per random record. (4) Cold pipeline — fetch + open + first field read. (5) Aggregate — sum score over all rows (including NXS WASM reducer when loaded). Use size presets or upload matching .nxb / .json / .csv files to substitute fixtures.

1. Open file — parse the entire structure

JSON and CSV eagerly parse every byte; NXS reads only the header + tail-index.

2. Random-access read — one field from record k

Once data is in memory, read record[k].username. Measures warm property access.

3. Random-access read — multiple fields from record k

Read 4 fields per random record (username, age, balance, active). NXS amortises header parsing after the first field.

4. Cold pipeline — fetch + parse + one field

The realistic browser scenario: user hits page, data fetched, first field read.

5. Aggregate — sum of 'score' across all records

Uses format-specific optimisations: JSON.parse + loop, raw CSV column scan, NXS bulk reducer (WASM if loaded).

6. WAL span ingestion — append · recover · seal · roundtrip

Simulated in-memory WAL path (no real disk I/O in browser). Encodes span structs into NXS binary objects and measures per-span cost at each pipeline stage. These numbers come from the Rust bench binary run locally — the browser renders the results as a reference chart.
